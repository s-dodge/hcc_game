PRONOUN REFERENCE ("take it", "read it", etc.)
- Track last interacted item in a `last_item` variable (initially None)
- After resolve_alias, check: if noun == "it" and last_item: noun = last_item
- Update last_item on successful examine and read interactions
- Do NOT update on take (item is in inventory, reference would be stale)
- Edge case: last_item should reset or warn if item is no longer in room/inventory

PARSER / ITEM ALIASES
- Add item aliases so multi-word items can be referenced by shorter names
- Example: "stick of ddr3 ram" should work with: "ram", "ddr3", "ddr3 ram", "stick"
- Implement as dict mapping aliases to canonical item names: {"ram": "stick of ddr3 ram", "ddr3": "stick of ddr3 ram", ...}
- Parser should check aliases first, then fall back to exact match
- Other examples: "bin of mice" → ["mice", "bin"], "broken keyboard" → ["keyboard"]
- Related: display_names dict maps canonical key → formatted display string (e.g. "usb drive" → "usb drive")
- These solve opposite problems but live together in items.py: aliases handle input, display_names handle output
- Pipeline: player input → aliases → canonical key → display_names → shown to player

SANITY EFFECTS
- Item descriptions change as sanity drops (examine checks sanity threshold, returns alternate description)
- inventory_description attribute on Item: short string shown in inventory table; separate from examine text
  - Follow same threshold dict pattern as sanity_descriptions — display_inventory resolves tier from current sanity
  - Three degradation axes: examine text (sanity_descriptions), inventory table description (inventory_description), item name corruption
  - Fallback: if inventory_description is None, use item.description in table
- Multiple tiers: normal → unsettling → wrong → incomprehensible
- Example: "bin of mice" examine text changes from "computer mice" to real mice at low sanity
- Items may disappear from or change in inventory as sanity drops
- Items may appear in rooms that were not previously there ("You see X...was that here before?")
- Rooms may appear in different places ("You have entered the AV Tech department...That's not right, AV Tech isn't in this building")
- Inventory corruption: items rename themselves at low sanity ("badge" → "a tooth", "ddr3" → "a finger bone")
- Item names in room display and inventory fragment via zalgo/unicode before descriptions go full eldritch
- Non-euclidian topology, eldrich runes, blood coverage as description tiers
- Consider a dict of sanity-gated alternate descriptions per item keyed by threshold, e.g. {75: "unsettling text", 50: "wrong text", 25: "incomprehensible text"}
- "book" (Windows 11: Inside Out) sanity tiers:
  - 100-76: "It's an untouched copy of 'Windows 11: Inside Out'." (normal)
  - 75-51: "The cover reads 'Windows 11: Inside Out'...but the pages seem to be written in Latin." (unsettling)
  - 50-26: "The title has changed. It now reads 'The Necronomicon: Inside Out'. The pages are warm to the touch." (wrong)
  - 25-0: Inventory name changes from "book" → "grimoire". "The grimoire writhes in your hands. The text is written in a language that predates human civilization. You understand every word." (incomprehensible)

USB DRIVE / WORKSTATION STATE (persistent content display)
- After `use drive on workstation`, drive should NOT be consumed — instead:
  1. Remove drive from player.inventory
  2. Add drive to room.items (IT dept room)
  3. Set drive.state = "inserted"
  4. Set workstation.state = "displaying"
- `examine workstation` (or `read workstation`) checks state == "displaying" and shows file contents
- `take drive` works normally (drive is a room item); in handle_take(), after taking it, check
  drive.state == "inserted" and reset workstation.state back to "powered"
- drive.state drives its in-room description (e.g. "a USB drive is plugged into the workstation")
- workstation description varies by state: off → powered on → displaying a file

THE INTERN
- The IT intern is a cultist (or was one)
- Their absence recontextualizes the whole IT building: badge left behind, broken workstation, USB drive, floppy disk in the closet
- The service ticket ("urgent password reset at the LRC") may be the intern luring IT staff to the library — or someone/something else sending the player there
- The intern is likely at the LRC (the main game destination)
- Key question to decide: true believer, victim, or someone who stumbled in and didn't get out?
  - True believer: badge left deliberately, clues are breadcrumbs laid on purpose
  - Victim: badge dropped in panic, clues are distress signals
  - Stumbled in: badge left innocently (forgot it), clues are accidental — scarier because it could happen to the player too
